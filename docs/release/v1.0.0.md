# arena-lang v1.0.0 — API freeze

**The surface is frozen.** v0.2.0 built the core; v1.0.0 commits to it. Every public
item — `Arena<T>`, `Id<T>`, and `ArenaError` — is now stable under Semantic
Versioning: within the `1.x` series nothing is removed or changed incompatibly, and a
breaking change waits for `2.0`. There is no functional change from v0.2.0. This tag
is the promise that code written against arena-lang today keeps compiling against
every `1.x` release, so a crate above it can depend on the handle's shape and the
allocation contract without watching for churn.

## What is arena-lang?

The allocation floor a tree of compiler nodes is built on. It hands out a value's
storage from a contiguous arena, returns a four-byte, type-tagged handle that never
goes stale for the life of the arena, and frees everything at once when the arena is
dropped. It is the base-tier allocation crate beneath `ast-lang` and the IR — nodes
allocated forward, handles that stay put as the arena grows, no per-node
deallocation. It owns typed allocation and stable addressing only.

## The frozen surface

The full public API, stable as of this tag:

- **`Arena<T>`** — the typed, append-only arena: `new`, `with_capacity`, `reserve`,
  `alloc`, `try_alloc`, `get`, `get_mut`, `contains`, `len`, `is_empty`, `capacity`,
  and `iter`. `Default` and `Debug` (which prints size, not contents, so it needs no
  `T: Debug`).
- **`Id<T>`** — a four-byte, `Copy`, type-tagged handle, `Eq` / `Ord` / `Hash` for
  every `T`, so it works directly as a map key. Stable for the life of the arena that
  minted it.
- **`ArenaError`** — `#[non_exhaustive]`; the `CapacityExhausted` variant is the
  defined outcome at the `u32::MAX`-value ceiling, returned by `try_alloc`.

What the freeze guarantees: the size and trait impls of `Id<T>`, the append-only
allocation model, that a handle survives arbitrary later growth, and that exhaustion
is a reported error rather than a panic (from `try_alloc`) or a silent wrap. `Arena`
keeps `#![forbid(unsafe_code)]` and zero first-party dependencies.

## Breaking changes

- **The `serde` feature and its optional dependency are removed.** It was a reserved
  no-op stub that derived nothing on any type. Rather than freeze a dead feature flag
  into the `1.x` contract, it is dropped; the only remaining feature is `std`
  (default, opt-out for `no_std`). Because the stub had no effect, no working code
  relied on it. `serde` support can return later as an additive, non-breaking minor
  release if it is ever needed.

## Verification

Tested on Linux (WSL2 Ubuntu, `cargo 1.95`); the same commands run across the CI
matrix on Linux, macOS, and Windows against both stable and the 1.85 MSRV:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
cargo build --no-default-features
cargo +1.85 build
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo audit
cargo deny check
```

All green. Counts at this tag: 12 unit + 4 property + 16 doctests. With `serde` gone,
`--all-features` is identical to the default build.

Benchmarks (`alloc` and `get`, on the reference machine, WSL2 Ubuntu), reported per
operation from batches of 1,024 to 262,144 nodes:

- **`alloc`** — about **0.4 ns per node**, flat from 1K to 262K nodes. With capacity
  reserved, allocation is a bump append: a write and a length bump, no per-node
  bookkeeping.
- **`get`** — about **0.4 ns per handle**, flat across the same range. Resolution is a
  single bounds-checked indexed read, constant time, no search.

Both costs are independent of how many nodes the arena already holds — the property
the contiguous-store, indexed-handle design exists to provide.

## What's next

1.0 is a stability boundary, not an end. The `1.x` series takes only additive,
non-breaking change: documentation, internal optimisation, and any genuinely useful
new method or trait impl that does not alter the frozen items. Anything that would
break the surface — a wider handle, a changed allocation model — waits for a
deliberate `2.0`.
